# ProcessCheck 3관점 기능 제안서

> **문서 버전**: 1.0
> **작성일**: 2026-03-11
> **목적**: 현재 시스템 분석을 바탕으로 3가지 관점에서 개선 방향을 제안한다.

---

## 관점 1: 사용자 관점 (실무자/매니저)

### 현재 UX 분석

**좋은 점**:
- 역할 기반 대시보드로 각 사용자에게 필요한 정보만 노출
- 실시간 Firestore 구독으로 변경 사항이 즉시 반영
- 7가지 뷰 모드(테이블/매트릭스/카드/간트/칸반/타임라인/캘린더)로 다양한 시각화 제공
- 통계 카드 클릭으로 해당 섹션/필터로 즉시 이동
- 한국어 UI로 도메인 전문가의 진입 장벽이 낮음

**불편한 점**:
- 작업 일괄 처리 기능 부재 (하나씩만 완료/승인 가능)
- 파일 업로드가 실제로 동작하지 않음
- 모바일에서의 복잡한 테이블/매트릭스 뷰 사용성 제한
- 담당자 배정이 수동 (텍스트 입력)
- 작업 간 의존성 시각화 부재

### 개선 제안

#### U-01. 작업 일괄 완료/승인 (Bulk Actions)
**설명**: 체크리스트 탭에서 여러 작업을 체크박스로 선택한 뒤 "일괄 완료" 또는 "일괄 승인" 버튼으로 한 번에 처리할 수 있게 한다. 매니저가 동일 부서의 여러 작업을 승인할 때 특히 유용하다.
**예상 효과**: 매니저의 승인 처리 시간 70% 이상 단축, 클릭 피로도 감소
**구현 난이도**: 하
**우선순위**: P0

#### U-02. 드래그앤드롭 작업 재배치 (Drag & Drop Reorder)
**설명**: 칸반 뷰에서 작업 카드를 드래그하여 상태를 변경하고, 체크리스트 목록에서 작업 순서를 드래그로 변경할 수 있게 한다. 칸반 열 간 이동으로 상태(pending -> in_progress -> completed)를 직관적으로 전환한다.
**예상 효과**: 칸반 뷰의 실용성 향상, 직관적인 상태 관리
**구현 난이도**: 중
**우선순위**: P1

#### U-03. 담당자 드롭다운 선택 (Assignee Picker)
**설명**: 작업의 담당자(assignee)와 검토자(reviewer) 필드를 텍스트 입력 대신 부서별 사용자 드롭다운으로 변경한다. Firestore의 users 컬렉션에서 부서별로 필터링하여 선택지를 제공한다.
**예상 효과**: 입력 오류 방지, 일관된 사용자명 사용, 배정 속도 향상
**구현 난이도**: 하
**우선순위**: P0

#### U-04. 마감일 알림 강화 (Enhanced Deadline Alerts)
**설명**: 마감 3일/1일/당일/초과에 따라 색상 단계적 경고를 표시하고, 브라우저 Push Notification을 지원한다. 대시보드에 "오늘 마감" / "지연 작업" 섹션을 별도로 분리하여 긴급 작업을 즉시 파악할 수 있게 한다.
**예상 효과**: 마감 지연 건수 30% 이상 감소, 긴급 작업 대응 속도 향상
**구현 난이도**: 중
**우선순위**: P1

#### U-05. 모바일 최적화 뷰 (Mobile-First Views)
**설명**: 모바일에서 복잡한 매트릭스/테이블 뷰 대신 카드 기반 축약 뷰를 기본으로 표시하고, 스와이프 제스처로 작업 완료/승인을 처리할 수 있게 한다. 네비게이션을 하단 탭 바로 전환한다.
**예상 효과**: 모바일 사용률 향상, 현장 작업자의 실시간 업데이트 가능
**구현 난이도**: 상
**우선순위**: P2

#### U-06. 작업 타이머/시간 추적 (Time Tracking)
**설명**: 각 작업에 소요 시간을 기록할 수 있는 타이머 기능을 추가한다. 시작/중지 버튼으로 실제 작업 시간을 측정하고, 예상 소요 시간 대비 실제 소요 시간을 비교할 수 있게 한다.
**예상 효과**: 프로젝트 일정 추정 정확도 향상, 업무 부하 파악
**구현 난이도**: 중
**우선순위**: P2

#### U-07. 작업 의존성 시각화 (Dependency Graph)
**설명**: 작업 간 의존 관계를 방향 그래프(DAG)로 시각화한다. 현재 `dependencies` 필드가 존재하지만 UI에서 활용되지 않고 있다. 선행 작업이 미완료일 때 경고를 표시하고, 크리티컬 패스를 강조한다.
**예상 효과**: 병목 지점 즉시 파악, 일정 지연 사전 예방
**구현 난이도**: 상
**우선순위**: P2

#### U-08. 개인 작업 캘린더 (Personal Calendar)
**설명**: 대시보드에 개인 할당 작업만 표시하는 캘린더 위젯을 추가한다. Google Calendar / Outlook과 iCal 형식으로 동기화할 수 있는 내보내기 기능도 함께 제공한다.
**예상 효과**: 개인 일정 관리 효율 향상, 외부 캘린더와의 통합
**구현 난이도**: 중
**우선순위**: P2

#### U-09. 작업 템플릿 즐겨찾기 (Quick Templates)
**설명**: 자주 생성하는 작업 유형을 즐겨찾기로 저장하여 한 번의 클릭으로 동일 구조의 작업을 생성할 수 있게 한다. 부서별 자주 사용하는 작업 패턴을 공유 가능하다.
**예상 효과**: 반복 작업 생성 시간 단축, 작업 표준화
**구현 난이도**: 하
**우선순위**: P3

#### U-10. 오프라인 모드 지원 (Offline Support)
**설명**: Service Worker를 활용하여 네트워크가 불안정한 환경에서도 최근 데이터를 캐시하여 조회 가능하게 하고, 오프라인 상태에서의 변경 사항을 온라인 복귀 시 자동 동기화한다. Firestore의 `enablePersistence()`를 활용한다.
**예상 효과**: 공장/현장에서의 안정적 사용, 네트워크 장애 시 업무 연속성 보장
**구현 난이도**: 상
**우선순위**: P3

## 관점 2: 관리자 관점 (기획조정실/시스템 관리자)

### 현재 관리 기능 분석

**좋은 점**:
- 기획조정실이 전체 프로젝트를 한눈에 모니터링 가능
- 6 Phase 매트릭스 뷰로 전사적 진행 상황 파악 용이
- 193개 표준 템플릿으로 프로세스 일관성 보장
- 자동 통계 재계산(progress/riskLevel/currentStage)

**불편한 점**:
- 대시보드 커스터마이징 불가 (고정된 카드 레이아웃)
- 정기 보고서 자동 생성 기능 없음
- 감사 로그/변경 이력 추적 없음
- SLA/KPI 추적 대시보드 없음
- 부서별 성과 비교 분석 없음

### 개선 제안

#### A-01. 커스터마이즈 가능한 대시보드 (Dashboard Builder)
**설명**: 기획조정실이 대시보드의 위젯을 자유롭게 배치/크기 조절/추가/제거할 수 있는 드래그앤드롭 대시보드 빌더를 제공한다. 위젯 종류: 프로젝트 진행률 차트, 부서별 작업 현황, 마감 경고, 승인 대기 큐, KPI 카드 등을 선택 가능하게 한다.
**예상 효과**: 관리자별 맞춤 정보 제공, 의사결정 속도 향상
**구현 난이도**: 상
**우선순위**: P2

#### A-02. 자동 정기 보고서 생성 (Auto Report Generation)
**설명**: 주간/월간 단위로 프로젝트 현황 보고서를 자동 생성하여 PDF 또는 이메일로 배포한다. 보고서에 포함할 항목: 프로젝트별 진행률, 지연 작업 목록, 승인 현황, 부서별 성과, 위험 프로젝트 경고.
**예상 효과**: 보고서 작성 시간 주당 2-3시간 절약, 일관된 보고 형식
**구현 난이도**: 중
**우선순위**: P1

#### A-03. 감사 로그 (Audit Trail)
**설명**: 모든 데이터 변경 사항을 타임스탬프, 변경자, 변경 전/후 값과 함께 별도 `auditLogs` 컬렉션에 기록한다. ISO 13485 규제 대응에 필수적인 추적성(traceability)을 보장한다. 관리자 페이지에서 검색/필터링 가능한 감사 로그 뷰어를 제공한다.
**예상 효과**: 규제 감사 대응 시간 대폭 단축, 변경 추적성 확보, ISO 13485 준수
**구현 난이도**: 중
**우선순위**: P0

#### A-04. SLA 추적 대시보드 (SLA Tracking)
**설명**: 각 Phase별 목표 기간과 실제 소요 기간을 비교하는 SLA 대시보드를 추가한다. Phase별 평균 처리 시간, 승인 대기 평균 시간, 지연율 트렌드를 시계열 차트로 표시한다.
**예상 효과**: 프로세스 병목 조기 발견, 일정 추정 정확도 향상
**구현 난이도**: 중
**우선순위**: P1

#### A-05. 부서별 성과 분석 (Department Analytics)
**설명**: 부서별 작업 완료율, 평균 처리 시간, 반려율, 마감 준수율을 비교하는 분석 대시보드를 제공한다. 기간별 필터링과 프로젝트별 필터링을 지원하고, 차트(막대/레이더/파이)로 시각화한다.
**예상 효과**: 부서별 개선 포인트 발견, 자원 배분 최적화
**구현 난이도**: 중
**우선순위**: P1

#### A-06. 프로젝트 비교 뷰 (Project Comparison)
**설명**: 2-3개 프로젝트를 나란히 놓고 Phase별 진행률, 일정 준수율, 부서별 완료율을 비교하는 뷰를 제공한다. 과거 완료 프로젝트와 현재 진행 프로젝트의 속도를 비교하여 지연 예측에 활용한다.
**예상 효과**: 프로젝트 간 벤치마킹, 지연 조기 경고
**구현 난이도**: 중
**우선순위**: P2

#### A-07. 자원 할당 뷰 (Resource Allocation View)
**설명**: 부서/개인별 현재 할당된 작업 수와 마감일을 히트맵으로 표시하여 과부하/유휴 상태를 시각화한다. 새 작업 배정 시 부하 균형을 고려한 담당자 추천 기능도 포함한다.
**예상 효과**: 작업 부하 균형 최적화, 담당자 배정 효율 향상
**구현 난이도**: 상
**우선순위**: P2

#### A-08. 이메일/Slack 연동 알림 (External Notification Integration)
**설명**: 현재 인앱 알림만 존재하는 것을 이메일 및 Slack/Teams 웹훅 연동으로 확장한다. 알림 종류별로 채널을 선택할 수 있게 하여 긴급 알림은 이메일+Slack, 일반 알림은 인앱만 등의 설정을 지원한다.
**예상 효과**: 알림 도달률 향상, 긴급 상황 대응 속도 개선
**구현 난이도**: 중
**우선순위**: P1

#### A-09. 권한 세분화 (Fine-grained Permissions)
**설명**: 현재 3단계(worker/manager/observer) 역할을 더 세분화하여 프로젝트별 역할 배정, 기능별 권한 매트릭스를 지원한다. "프로젝트 A에서는 매니저, 프로젝트 B에서는 실무자" 같은 복합 역할을 허용한다.
**예상 효과**: 유연한 조직 구조 대응, 최소 권한 원칙 적용
**구현 난이도**: 상
**우선순위**: P2

#### A-10. 데이터 백업/복원 (Backup & Restore)
**설명**: Firestore 데이터의 정기 백업(일/주 단위)을 Firebase Cloud Functions로 자동화하고, 관리자가 특정 시점으로 데이터를 복원할 수 있는 UI를 제공한다. 실수로 삭제된 데이터 복구에 필수적이다.
**예상 효과**: 데이터 손실 방지, 장애 복구 시간 단축
**구현 난이도**: 중
**우선순위**: P1

## 관점 3: 시스템 관점 (개발자/아키텍트)

### 현재 아키텍처 분석

**강점**:
- Client-Side Only 아키텍처로 서버 관리 부담 없음
- Firebase 실시간 구독으로 별도 WebSocket 서버 불필요
- Static Export로 GitHub Pages에 무료 배포
- HTML Port와 Next.js App의 이중 포트로 빌드 실패 시에도 서비스 가능
- 193개 표준 템플릿 기반 자동 체크리스트 생성

**약점**:
- 테스트 프레임워크 부재 (유닛/E2E 테스트 없음)
- Firestore 보안 규칙 미정의 (클라이언트에서 직접 DB 접근)
- HTML Port와 Next.js App 간 수동 동기화 필요
- 에러 모니터링/로깅 시스템 없음
- 번들 사이즈 최적화 미적용
- 환경별(dev/staging/prod) 배포 파이프라인 없음

### 개선 제안

#### S-01. 테스트 프레임워크 도입 (Testing Framework)
**설명**: Vitest + React Testing Library로 유닛 테스트를 구성하고, Playwright로 E2E 테스트를 추가한다. 핵심 비즈니스 로직(승인 워크플로우, 템플릿 필터링, 프로젝트 통계 계산)부터 테스트를 작성하고, CI에서 자동 실행되도록 GitHub Actions에 통합한다.
**예상 효과**: 회귀 버그 방지, 리팩토링 안전성 확보, 코드 품질 향상
**구현 난이도**: 중
**우선순위**: P0

#### S-02. Firestore 보안 규칙 정의 (Security Rules)
**설명**: 현재 클라이언트에서 Firestore에 직접 접근하면서 보안 규칙이 정의되어 있지 않다. 역할 기반 접근 제어를 Firestore Security Rules로 구현한다: users 컬렉션은 본인만 수정, checklistItems는 assignee/reviewer만 상태 변경, templateItems는 manager/observer만 수정 등.
**예상 효과**: 데이터 무결성 보장, 무단 접근 방지, 프로덕션 보안 수준 달성
**구현 난이도**: 중
**우선순위**: P0

#### S-03. 에러 모니터링 도입 (Error Monitoring)
**설명**: Sentry 또는 Firebase Crashlytics를 도입하여 클라이언트 사이드 에러를 자동 수집하고, 슬랙 채널로 실시간 알림을 보낸다. Firestore 쿼리 실패, 인증 오류, UI 렌더링 에러를 분류하여 추적한다.
**예상 효과**: 프로덕션 버그 조기 발견, MTTR(Mean Time To Resolve) 단축
**구현 난이도**: 하
**우선순위**: P0

#### S-04. CI/CD 파이프라인 강화 (Enhanced CI/CD)
**설명**: 현재 main push 시 자동 배포만 되는 파이프라인을 확장한다. PR 생성 시 린트+타입체크+유닛테스트 자동 실행, Lighthouse CI로 성능 점수 체크, Preview 배포로 코드 리뷰 시 실제 화면 확인 가능하게 한다. dev/staging/production 3단계 환경을 분리한다.
**예상 효과**: 배포 안전성 향상, 코드 품질 게이트 확보
**구현 난이도**: 중
**우선순위**: P1

#### S-05. HTML Port 자동 동기화 (Auto-Sync Tooling)
**설명**: Next.js 앱의 firestoreService.ts 변경 시 HTML Port의 firestore-service.js와의 차이를 자동 감지하는 CI 체크를 추가한다. AST 기반 함수 시그니처 비교 도구를 개발하여 함수 추가/삭제/변경을 자동으로 리포트한다.
**예상 효과**: 동기화 누락 방지, 두 포트 간 일관성 보장
**구현 난이도**: 상
**우선순위**: P1

#### S-06. 성능 최적화 (Performance Optimization)
**설명**: Firestore 쿼리 최적화(복합 인덱스 활용, 페이지네이션 도입), React 컴포넌트 메모이제이션(`React.memo`, `useMemo`), 코드 스플리팅(dynamic import), 이미지 lazy loading을 적용한다. 특히 프로젝트 목록 페이지에서 전체 checklistItems를 로드하는 현재 방식을 개선한다.
**예상 효과**: 초기 로딩 시간 50% 단축, Firestore 읽기 비용 절감
**구현 난이도**: 중
**우선순위**: P1

#### S-07. TypeScript 타입 안전성 강화 (Type Safety)
**설명**: 현재 HTML Port가 Vanilla JS로 작성되어 타입 안전성이 없다. JSDoc 타입 주석을 체계적으로 추가하거나, HTML Port도 TypeScript로 마이그레이션한다. Firestore 데이터 변환 함수에 제네릭 타입을 적용하여 런타임 타입 오류를 컴파일 타임에 잡는다.
**예상 효과**: 런타임 에러 감소, IDE 자동완성 지원, 개발 생산성 향상
**구현 난이도**: 중
**우선순위**: P2

#### S-08. Firebase Cloud Functions 도입 (Server-Side Logic)
**설명**: 현재 클라이언트에서 수행하는 복잡한 비즈니스 로직(프로젝트 통계 재계산, 알림 생성, 포털 알림)을 Firebase Cloud Functions로 이전한다. Firestore Trigger를 활용하여 문서 변경 시 자동으로 서버 사이드에서 처리한다.
**예상 효과**: 클라이언트 부하 감소, 비즈니스 로직 중앙화, 보안 강화
**구현 난이도**: 상
**우선순위**: P2

#### S-09. 모노레포 구조 전환 (Monorepo Architecture)
**설명**: 현재 Next.js App과 HTML Port가 하나의 레포에 혼재되어 있다. Turborepo 또는 Nx로 모노레포 구조를 전환하여 공유 로직(Firestore 서비스, 타입, 유틸리티)을 패키지로 분리하고, 각 앱(nextjs-app, html-port)이 공유 패키지를 참조하도록 한다.
**예상 효과**: 코드 중복 제거, 동기화 문제 근본 해결, 빌드 캐싱 활용
**구현 난이도**: 상
**우선순위**: P3

#### S-10. API 추상화 레이어 도입 (API Abstraction Layer)
**설명**: Firestore에 직접 의존하는 현재 구조에서, Repository Pattern 기반 추상화 레이어를 도입한다. `IProjectRepository`, `IChecklistRepository` 등의 인터페이스를 정의하고, Firestore 구현체를 주입하는 방식으로 변경한다. 이를 통해 향후 백엔드 변경(Firestore -> PostgreSQL 등)이나 테스트 시 Mock 주입이 용이해진다.
**예상 효과**: 백엔드 교체 용이성, 테스트 가능성 향상, 관심사 분리
**구현 난이도**: 상
**우선순위**: P3
