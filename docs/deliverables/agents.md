# ProcessCheck 에이전트 정의서

프로젝트 산출물 생성을 위한 에이전트 유형별 명세. 총 12개 에이전트.

---

## 기존 에이전트 (1-6)

---

### /diagram — 구조도 에이전트

**산출물**: `diagrams.drawio` (단일 파일, 멀티탭) + `diagram-viewer.html` (브라우저 뷰어)

| 탭 | 내용 |
|---|---|
| 사이트맵 | 전체 페이지 구조, 네비게이션 관계, 인증 필요 여부, 역할별 접근 권한 |
| 데이터 모델 | Firestore 10개 컬렉션 ERD, 필드 정의, 1:N 관계선, 도메인별 색상 |
| 프로세스 맵 | 6 Phase 승인 워크플로우 swimlane, 작업→매니저승인→게이트승인, 설계변경 3경로 |
| 시스템 아키텍처 | 3계층(클라이언트/Firebase/배포), 인증 2경로, 데이터 흐름 |

**형식 규칙**:
- draw.io XML 형식 (.drawio)
- 한 파일에 모든 다이어그램을 탭으로 분리
- HTML 뷰어로 브라우저에서 바로 확인 가능
- draw.io 앱에서도 열기/편집 가능

**활용 시나리오**: 시스템 전체 구조를 시각적으로 파악해야 할 때, 신규 팀원 온보딩 시, 아키텍처 리뷰 회의에서 활용. `/handoff` 에이전트의 아키텍처 오버뷰 섹션과 연계.

---

### /screen — 화면 에이전트

**산출물**: `wireframes.html` (단일 HTML, 전체 화면 한판 보기)

**포함 화면 (10개)**:
1. 로그인 — 데모카드 3개 + Microsoft OAuth
2. 대시보드 — 통계카드 5개 + 작업/프로젝트/알림 테이블
3. 프로젝트 목록 — 7개 뷰(테이블/매트릭스/카드/간트/칸반/타임라인/캘린더)
4. 프로젝트 상세 — 4개 탭(개요/체크리스트/설계변경/출시준비)
5. 작업 상세 — 2컬럼(정보+타임라인), 역할별 액션 버튼
6. 체크리스트 관리 — 3개 뷰(매트릭스/트리/리스트)
7. 고객 관리 — 검색/필터 + 고객 테이블 + 상세 패널
8. 고객 포털 — 이메일 인증 + 읽기전용 현황
9. 영업 대시보드 — D-Day, 출시 준비 진행률
10. 매뉴얼 — 사이드바 TOC + 콘텐츠

**디테일 규칙**:
- 모든 버튼, 입력필드, 뱃지, 테이블 컬럼을 **실제 구현과 동일하게** 그리기
- 한국어 레이블 + 실제 샘플 데이터
- 브라우저 프레임으로 각 화면 감싸기
- 어노테이션 번호 + 설명 목록
- 클릭 가능 요소 hover 효과
- **한 장에 전부** — 스크롤로 전체 화면 순회, 상단 고정 네비게이션 점프

**활용 시나리오**: 기획 단계에서 UI 레이아웃 합의, 이해관계자 리뷰 시 실제 화면 미리보기 제공, `/spec` 에이전트의 페이지별 기능 명세와 1:1 대응.

---

### /spec — 스펙 에이전트

**산출물**: `spec.md` (개발자용 기능 명세서)

**구성**:
1. 시스템 개요 (목적, 대상 사용자)
2. 기술 스택 상세
3. 인증 체계 (데모 + OAuth)
4. 데이터 모델 (필드별 타입/설명/필수여부)
5. 서비스 함수 목록 (firestoreService 전체 함수, 파라미터, 반환값)
6. 페이지별 기능 명세 (데이터 소스, 인터랙션, 권한)
7. 역할별 권한 매트릭스 (CRUD)
8. 승인 워크플로우 상세 (work stage vs gate stage)
9. 설계변경 프로세스 (유형별 단계, 필터링 규칙)
10. 알림 체계 (이벤트→대상→내용 매핑)
11. Known Gaps & TODO

**활용 시나리오**: 개발자가 구현 시 참조하는 1차 문서, QA 테스트 케이스 작성의 기준, `/handoff` 에이전트와 함께 전달하여 완전한 개발 패키지 구성.

---

### /ideate — 아이디어 에이전트

**산출물**: `ideate.md` (3관점 기능 제안서)

**관점 3가지**:
1. **사용자 관점** (실무자/매니저) — UX 개선 제안 10가지
2. **관리자 관점** (기획조정실/시스템관리자) — 관리 기능 개선 10가지
3. **시스템 관점** (개발자/아키텍트) — 기술 개선 10가지

각 제안: 제목, 설명(2-3문장), 예상 효과, 구현 난이도(상/중/하), 우선순위(P0-P3)

**활용 시나리오**: 스프린트 백로그 작성 시 아이디어 소스, 로드맵 수립 시 기능 후보 목록, `/matrix` 에이전트로 우선순위 매트릭스 생성 후 실행 순서 결정.

---

### /interview — 인터뷰 에이전트

**산출물**: `interview-notes.md` (미팅노트 → 요구사항 추출)

**프로세스**:
1. 미팅노트/대화록 입력 받기
2. 핵심 요구사항 추출 (기능/비기능 분류)
3. 우선순위 매기기 (MoSCoW: Must/Should/Could/Won't)
4. 현재 시스템과의 Gap 분석
5. 액션 아이템 도출

**활용 시나리오**: 고객 미팅, 내부 기획 회의, 이해관계자 인터뷰 후 결과물 정리. `/action-plan` 에이전트로 액션 아이템을 실행 계획으로 전환 가능.

---

### /handoff — 핸드오프 에이전트

**산출물**: `handoff.md` (개발자 패키지)

**구성**:
1. 프로젝트 온보딩 가이드 (환경 셋업 순서)
2. 아키텍처 오버뷰 (폴더 구조, 데이터 흐름)
3. 코딩 컨벤션
4. **산출물 인덱스** (전체 deliverables 파일 목록 + 설명)
5. 주요 의사결정 로그
6. 개발 체크리스트
7. 리소스 링크

**활용 시나리오**: 신규 개발자 합류 시 1차 전달 문서, 외부 개발팀 협업 시 프로젝트 컨텍스트 전달, `/diagram` + `/spec` + `/screen` 산출물과 함께 완전한 핸드오프 패키지 구성.

## 신규 에이전트 (7-12)

---

### /extract — 문서 인사이트 추출 에이전트

**산출물**: `findings.md`

**입력**: PDF, Word, 웹페이지, 회의록 등 모든 문서

**프로세스**:
1. 문서를 섹션별로 분석
2. 핵심 발견(Findings) 추출 — 사실/데이터 기반
3. 추천사항(Recommendations) 추출 — 행동 제안
4. 인용/출처 번호 보존 [123] 형식
5. 테마별 그룹핑 (예: 채용, 관계구축, 교육...)

**출력 구조**:
```markdown
# 문서 인사이트 추출 결과
## 문서 정보
- 제목, 저자, 일자, 분량
## 핵심 발견 (N개)
### 테마 1: [테마명]
- **발견**: [내용] [출처]
- **시사점**: [해석]
### 테마 2: ...
## 추천사항 요약 (N개)
- 즉시 실행 가능 (Quick Wins)
- 중기 과제
- 장기 전략
## 데이터 포인트
- 주요 수치/통계 목록
```

**활용 시나리오**: 경쟁사 보고서, 기술 논문, 컨설팅 보고서 등 대량 문서에서 핵심만 빠르게 추출. `/action-plan` 에이전트의 직접적인 입력 소스.

---

### /action-plan — 액션 플랜 에이전트

**산출물**: `action-plan.html` (인터랙티브 HTML)

**입력**: findings.md 또는 원본 문서

**핵심 포맷**: "왜→어떻게→실행 체크리스트" 패턴을 자동 생성

**각 액션 아이템 구조**:
```
[액션 아이템 제목]
+---------------------------------------------+
| 왜 해야 하나요? (Why)                         |
| - 근거 1 [출처]                              |
| - 근거 2 [출처]                              |
| - 근거 3 [출처]                              |
+---------------------------------------------+
| 어떻게 하나요? (How)                          |
| 1. 단계 1: [상세 설명] [출처]                 |
| 2. 단계 2: [상세 설명] [출처]                 |
| 3. 단계 3: [상세 설명] [출처]                 |
+---------------------------------------------+
| 실행 체크리스트                                |
| [ ] 체크항목 1                               |
| [ ] 체크항목 2                               |
| [ ] 체크항목 3                               |
+---------------------------------------------+
```

**HTML 기능**:
- 체크박스 클릭 → 진행률 바 업데이트
- localStorage에 상태 저장
- 담당자 배정 드롭다운
- 마감일 설정
- 우선순위 태그 (P0-P3)
- 플로우 다이어그램 (액션 간 의존성)
- "마크다운 복사" + "JSON 내보내기" 버튼
- 인쇄 친화적

**활용 시나리오**: 문서 분석 결과를 즉시 실행 가능한 계획으로 전환. 팀 회의에서 바로 담당자 배정 및 마감일 설정. `/tracker` 에이전트로 진행 상황 추적 연계.

---

### /tracker — 진행 추적 대시보드 에이전트

**산출물**: `tracker.html`

**입력**: action-plan의 액션 아이템 목록

**기능**:
- 전체 진행률 게이지
- 액션별 상태 (미시작/진행중/완료/차단됨)
- 담당자별 워크로드 차트
- 마감일 타임라인 (간트 형태)
- 주간/월간 리뷰 체크포인트
- 리스크 플래그 (지연 항목 하이라이트)
- CSV/JSON 내보내기

**활용 시나리오**: 주간 팀 미팅에서 진행 상황 리뷰, PM이 전체 액션 아이템의 상태를 한눈에 파악, 지연 항목 조기 식별 및 대응. `/brief` 에이전트로 경영진 보고용 요약 생성.

---

### /brief — 경영진 브리프 에이전트

**산출물**: `brief.html` (1-2페이지 요약)

**입력**: 긴 문서 (PDF, 보고서, 논문 등)

**구성**:
1. 한 줄 핵심 메시지 (30자 이내)
2. 핵심 숫자 3개 (KPI/데이터 포인트)
3. 상황 요약 (3-5문장)
4. 즉시 실행 TOP 3 (가장 중요한 액션)
5. 리스크 & 기회
6. 다음 단계 (Next Steps)

**스타일**: 경영진이 2분 안에 파악할 수 있는 깔끔한 레이아웃

**활용 시나리오**: 임원 보고, 이사회 자료, 투자자 브리핑 등 의사결정권자에게 핵심만 전달해야 할 때. 긴 보고서를 `/extract`로 분석한 후 `/brief`로 요약하는 파이프라인 활용.

---

### /matrix — 우선순위 매트릭스 에이전트

**산출물**: `priority-matrix.html`

**입력**: 액션 아이템 목록

**기능**:
- Impact x Effort 2x2 매트릭스 시각화
- 드래그 & 드롭으로 아이템 배치 가능
- 4사분면: Quick Wins / Major Projects / Fill-Ins / Thankless Tasks
- MoSCoW 자동 분류 (Must/Should/Could/Won't)
- 가중 점수 계산 (Impact 1-5 x Urgency 1-5 / Effort 1-5)
- 정렬: 점수순, 카테고리순, 담당자순
- localStorage 저장

**활용 시나리오**: 스프린트 계획 시 백로그 우선순위 결정, 리소스 제약 상황에서 최적 실행 순서 도출, `/ideate` 에이전트의 제안 30개를 우선순위별로 분류.

---

### /persona — 이해관계자 페르소나 에이전트

**산출물**: `personas.html`

**입력**: 프로젝트 맥락, 사용자 조사 데이터

**각 페르소나 카드**:
- 이름, 직함, 부서
- 아바타 (CSS 이니셜)
- "하루 일과" 타임라인
- 목표 (Goals) 3가지
- 불만사항 (Pain Points) 3가지
- 기대 (Expectations) 3가지
- 시스템 사용 빈도/패턴
- 인용구 (예: "가장 답답한 건 승인이 어디서 막혀있는지 모르는 것")

**활용 시나리오**: UX 설계 시 사용자 공감 형성, 기능 우선순위 결정 시 "이 기능이 누구에게 가장 필요한가?" 판단 근거, `/screen` 에이전트로 페르소나 기반 화면 설계.

## 에이전트 조합 파이프라인

에이전트를 순차적으로 조합하여 end-to-end 워크플로우를 구성할 수 있다.

---

### 파이프라인 1: 문서 → 실행 (Document to Action)

원본 문서(PDF, Word 등)에서 시작하여 실행 가능한 계획을 수립하고 추적하는 전체 흐름.

```
원본 문서 (PDF/Doc)
  → /extract (인사이트 추출)        → findings.md
  → /action-plan (액션 플랜 생성)   → action-plan.html
  → /matrix (우선순위 매트릭스)      → priority-matrix.html
  → /tracker (진행 추적)            → tracker.html
  → /brief (경영진 보고)            → brief.html
```

**사용 예시**: 컨설팅 보고서를 받아 핵심 내용을 추출하고, 액션 아이템을 도출한 뒤, 우선순위를 매기고, 진행 상황을 추적하며, 경영진에게 요약 보고.

---

### 파이프라인 2: 기획 → 개발 (Planning to Development)

요구사항이나 아이디어에서 시작하여 개발 가능한 산출물 패키지를 완성하는 흐름.

```
요구사항/아이디어
  → /ideate (기능 제안)             → ideate.md
  → /persona (사용자 페르소나)       → personas.html
  → /diagram (시스템 설계)           → diagrams.drawio + diagram-viewer.html
  → /screen (UI 프로토타입)          → wireframes.html
  → /spec (개발 명세)               → spec.md
  → /handoff (개발자 전달)           → handoff.md
```

**사용 예시**: 신규 기능 아이디어를 다각도로 검토하고, 대상 사용자를 정의한 뒤, 시스템 구조와 UI를 설계하고, 개발 명세를 작성하여 개발팀에 전달.

---

### 파이프라인 3: 미팅 → 추적 (Meeting to Tracking)

미팅 결과에서 즉시 액션 아이템을 도출하고 추적까지 연결하는 가벼운 흐름.

```
미팅 노트
  → /interview (요구사항 추출)       → interview-notes.md
  → /action-plan (액션 아이템)       → action-plan.html
  → /tracker (진행 추적)            → tracker.html
```

**사용 예시**: 고객 미팅 후 회의록에서 요구사항을 구조화하고, 바로 액션 플랜을 생성하여 담당자를 배정하고, 다음 미팅까지 진행 상황을 추적.
