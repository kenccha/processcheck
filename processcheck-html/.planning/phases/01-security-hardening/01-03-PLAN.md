<objective>
Audit innerHTML usages in js/components.js and ensure all user-controlled data is escaped with
escapeHtml() before injection into the DOM.

Purpose: The notification panel in components.js (renderNotifPanel) renders `n.title` and
`n.message` directly from Firestore notification documents. If a malicious actor writes a
notification with XSS payload as the title or message, it would execute in any user's browser
when the notification panel opens. escapeHtml() is already imported in components.js and used
for `user.name` — the same protection must apply to notification content.

Output: Updated js/components.js with escapeHtml() applied to all user-data innerHTML injections
in the notification panel and any other template literals that interpolate Firestore data.
</objective>

export function escapeHtml(str) {
  if (!str) return "";
  const div = document.createElement("div");
  div.textContent = str;
  return div.innerHTML;
}

    Apply escapeHtml() to user-controlled fields:
    ```javascript
    notifications.map(n => `
      <div class="notif-item${n.read ? "" : " unread"}" data-notif-id="${escapeHtml(n.id)}" data-notif-link="${escapeHtml(n.link || "")}">
        <div class="notif-item-title">${escapeHtml(n.title)}</div>
        <div class="notif-item-message">${escapeHtml(n.message)}</div>
        <div class="notif-item-time">${timeAgo(n.createdAt)}</div>
      </div>
    `)
    ```

    Rationale for each field:
    - n.id: Firestore auto-ID (alphanumeric, safe) — escapeHtml is zero-cost here, adds defense in depth
    - n.link: URL string — in an HTML attribute; escaping prevents attribute injection
    - n.title: user-originated string — MUST escape (direct XSS risk)
    - n.message: user-originated string — MUST escape (direct XSS risk)
    - timeAgo(n.createdAt): returns a formatted time string from a Date — safe, no user data

    After making this change, scan the REST of components.js for any other template literals that
